# Introducing Fluxion: State Management That Gets Out of Your Way

State management in modern TypeScript applications has a way of accumulating ceremony. Between action creators, reducers, selectors, and the glue code that wires them together, a feature that should take an afternoon can sprawl into a maze of boilerplate. **Fluxion** is a new state-management library built to reverse that trend. It pairs a tiny, fully type-inferred core with a reactive update model, so the shape of your store *is* its type — no manual annotations, no duplicated definitions, and no guessing what a piece of state looks like three files away. The result is code that reads like the data it manages, with the compiler quietly guaranteeing it stays correct as your app grows.

The benefits show up the moment you start writing real features. Because Fluxion infers types end to end, autocomplete works everywhere — in your stores, your derived values, and your component bindings — and a mistyped property name is a build error rather than a runtime surprise. Updates are fine-grained: when one slice of state changes, only the subscribers that actually read it re-render, which keeps large UIs responsive without manual memoization. State is immutable under the hood but written with ordinary mutable syntax, so you get predictable, traceable updates without the spread-operator gymnastics. And the whole core is a few kilobytes with zero runtime dependencies, so adoption never means shipping a heavier bundle to your users.

Those traits make Fluxion a natural fit across a wide range of projects. It excels in component-heavy single-page apps where shared state — auth sessions, feature flags, shopping carts — needs to flow cleanly between distant parts of the tree. Its derived-value system suits dashboards and data-visualization tools that compute expensive aggregations from raw inputs and want them cached automatically. Form-heavy apps benefit from the granular subscriptions that keep a single field's keystrokes from re-rendering the entire page. And because the core is framework-agnostic with thin adapters for React, Vue, and Svelte, teams running more than one framework — or migrating between them — can share a single source of truth instead of maintaining parallel state layers.

Getting started takes minutes. Install the package with `npm install fluxion`, then define a store by handing `createStore` an initial object; its type is inferred for you. Read values with `store.get()`, update them inside `store.set(state => { ... })` using plain assignments, and derive computed values with `store.derive(state => ...)`. In a React project, the `useStore` hook subscribes a component to exactly the fields it reads, and everything else is handled for you. From there, the [official documentation](https://fluxion.dev/docs) walks through async actions, persistence, and devtools integration. Give Fluxion a try on your next feature — you may find that the best state management is the kind you barely notice.
